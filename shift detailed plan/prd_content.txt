Bloomberg Terminal users trust the interface because it shows maximum data without clutter. For Motorway Oil Enterprise, this means: Tight line heights in data tables (44px row height maximum) Condensed but readable typography at all data zoom levels KPI cards that show value + trend + delta in one compact unit Sidebar that collapses to icon-only without losing functionality Dashboard grids using 12-column layouts, never wasted whitespace Pillar 2: Precision Color Coding Color is not decoration — it is data. Every color choice must have semantic meaning: Blue (#2563EB) — Primary data, active states, selected items Emerald (#059669) — Positive values, profits, successful operations Rose (#E11D48) — Negative values, losses, alerts, critical states Amber (#D97706) — Warnings, pending items, medium-priority alerts Cyan (#0EA5E9) — Live/real-time data, streaming indicators Violet (#7C3AED) — Premium features, reports, enterprise data Pillar 3: Micro-Interaction Richness The interface must feel alive. Users should receive constant, subtle feedback that their actions are registered and the system is responsive: Hover states on every interactive element (&lt; 150ms response) Active/pressed states with scale(0.97) transform Loading states with skeleton shimmer, not blank screens Success states with brief green pulse animation Error states with red shake animation (accessibility-safe) Number counters that animate from 0 to value on mount Chart data that draws progressively, not instantly Core Aesthetic Equation ℹ INFO DARK MODE (default): Deep Obsidian (#060B18) base + translucent glass cards + vivid accent colors + subtle glow shadows. LIGHT MODE: Crisp white (#F8FAFC) base + frosted glass surfaces + muted accents + clean drop shadows. BLOOMBERG PRO MODE: Pure black (#000000) base + amber/cyan data colors + monospace font for all data. 2.2 — Design System Hierarchy Layer What It Controls Implementation Location Foundation CSS variables, color tokens, spacing scale, z-index system index.css :root + .dark + .bloomberg Extension Tailwind custom colors, shadows, animations, keyframes tailwind.config.js theme.extend Primitives Base HTML elements: body, scrollbars, selection color, focus rings index.css @layer base Utilities Glass, gradient, glow, shimmer, mesh utility classes index.css @layer utilities Components Card, Button, Input, Badge, Chip base styles index.css @layer components Framer Motion Page transitions, sidebar animations, micro-interactions Layout.tsx, Sidebar.tsx, all interactive components Local Styles Component-specific styles using Tailwind className Each .tsx file directly 2.3 — Visual Inspiration References Product What We Borrow What We Avoid Bloomberg Terminal Information density, semantic color coding, data precision Outdated typography, keypad-focused UX, no touch support Linear.app Ultra-fast dark mode, tight spacing, keyboard navigation feel Over-minimalism that hides important data Apple macOS (Ventura+) Glassmorphism, frosted panels, smooth spring animations Hiding functionality behind excessive animations Vercel Dashboard Clean deployment UI, status indicators, real-time streaming feel Over-reliance on white space that wastes screen real estate Figma App Panel-based layout, resizable sidebars, professional tool feel Complexity appropriate for design tools but not data entry Stripe Dashboard Financial data presentation, subtle color coding, clean tables SaaS-focused patterns that don&apos;t fit petroleum operations SECTION 3 Design Token System — Colors, Typography, Spacing, Motion Design Token System 3.1 — Complete Color Token Specification Every color used in the application must come from this token table. The following rules define the correct and incorrect use of glass effects throughout the application. Rule Correct Use Incorrect Use Backdrop Filter backdrop-filter: blur(20px) on floating surfaces (cards, sidebar, modals, header) blur() on every element — excessive blur destroys contrast and readability Background Color rgba(10,22,40,0.75) dark / rgba(255,255,255,0.75) light — 75% opacity minimum 100% opacity (no glass effect) or &lt;50% (too transparent, unreadable) Border 1px solid rgba(255,255,255,0.08) in dark / rgba(0,0,0,0.06) in light Hard solid borders that break the glass illusion Layering 2–3 glass layers maximum: app bg → card → elevated card More than 3 glass layers — visual noise, performance cost Background Behind Glass Always have a rich gradient or mesh behind glass elements (glass is transparent) Glass on plain flat color — no transparency effect visible Text on Glass Ensure minimum 4.5:1 contrast ratio — add text-shadow if needed Light text on light glass or dark text on dark glass — unreadable Performance Use will-change: transform on glass elements that animate backdrop-filter on hundreds of small elements — destroys GPU performance 12.2 — Glass Layer Stack Definition Layer Element Examples Background Blur Border Shadow Layer 0 — Root App background, mesh gradient var(--bg-primary) + mesh overlay none none none Layer 1 — Surface Main cards, dashboard panels, sidebar var(--bg-elevated) 75–80% opacity blur(20px) 1px rgba(255,255,255,0.08) 0 4px 24px rgba(0,0,0,0.4) Layer 2 — Elevated Dropdowns, tooltips, context menus var(--bg-elevated) 85% opacity blur(32px) 1px rgba(255,255,255,0.12) 0 8px 40px rgba(0,0,0,0.5) Layer 3 — Modal Modals, full-screen drawers, side sheets var(--bg-elevated) 92% opacity blur(40px) 1px rgba(255,255,255,0.15) 0 20px 60px rgba(0,0,0,0.6) Backdrop Modal/drawer overlay rgba(0,0,0,0.5) blur(4px) none none SECTION 13 New Component Library Requirements New Component Library Requirements The following new components must be created as part of the v4.0 overhaul.
